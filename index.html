04DSL
module 1 · strategy dsl

낙주매매 v1 — 파라미터화 — dummy_strategy_v1.yaml

# ─ schema ────────────────────────────────────── schema_version: "1.0.0" meta: name: "낙주매매 v1 (코인)" notes: "EMA200 닿음 + 장봉 위치 + [15]볼 좁/돌파" universe: symbols: ["BTCUSDT", "ETHUSDT", ...] selection: mode: "volume_rank" top_n: 100 # 거래대금 상위 100 timeframes: ["1m", "15m", "1h", "1d"] # ── 사용자가 자유롭게 ±N% 조정할 파라미터 ── params: ema_period: 200 ema_touch_band: 0.5 # ±0.5% long_candle_pos_min: 30 # 장봉 30% 지점 long_candle_pos_max: 70 # 장봉 70% 지점 bb_15m_period: 20 bb_15m_stddev: 2 bb_15m_squeeze_pct: 1.1 # [15]볼폭 좁힘 기준 bb_24m_squeeze_pct: 2.2 # 24봉 bb_60m_squeeze_pct: 5.0 # [60]볼폭 5% pullback_days: 5 # x일 후 매수 yellow_bar_threshold_pct: 4 # 노랑장봉 정의(몸통 ≥ 4%) target_profit_pct: 4 # 1차 목표 stoploss_pct: 1.5 # [15]볼하 손절 rules: entry: op: "AND" conds: - { tf: "1d", lhs: "close", op: "near", rhs: "EMA(200)", band_pct: $ema_touch_band } - { tf: "1d", lhs: "yellow_bar_pos_pct", op: "between", lo: $long_candle_pos_min, hi: $long_candle_pos_max } - { tf: "15m", lhs: "BB_width_pct(20,2)", op: "<=", rhs: $bb_15m_squeeze_pct } - { tf: "1d", lhs: "quote_volume_rank", op: "<=", rhs: 100 } exit: op: "OR" conds: - { type: "take_profit_pct", value: $target_profit_pct } - { tf: "15m", lhs: "close", op: "<", rhs: "BB_lower(20,2)" } - { type: "stoploss_pct", value: $stoploss_pct } position: splits: # 분할매수 — §7 참고 - { weight: 0.40, trigger: "entry" } - { weight: 0.30, trigger: "drop_pct: 2" } - { weight: 0.30, trigger: "drop_pct: 4" }

→ M6 Optimizer가 params만 가지고 target_profit_pct ∈ [3, 8], stoploss_pct ∈ [1, 3] 같은 범위를 Optuna로 sweep합니다. 이게 "±N% 너 마음대로 변경"의 실현 방법.

i.
노랑장봉 — yellow_bar_pos_pct 계산법
몸통이 4% 이상인 양봉을 "노랑장봉"이라 정의. pos_pct = (close - open) / (high - low) × 100. 30~70% 구간이면 "몸통이 봉 중앙에 있음" → 매수자/매도자 균형 → 다음 캔들 변동성 폭발 확률 ↑. 0% 또는 100%면 한 방향 압도라 추세 전환 시그널이 아님.
indicator ▸ yellow bar
ii.
BB 폭 1.1% 의미 — 변동성 압축
BB_width = (upper - lower) / mid × 100. 평상시 BTC 15m 볼린저 폭은 1.5~3%, 1.1% 이하로 떨어지면 직전 ~30봉 대비 변동성 하위 10%. 압축 후 평균 8~12봉 안에 한 방향으로 폭발 — Bollinger Squeeze 페이퍼(2007) 기반의 검증된 패턴.
stat ▸ p10 volatility
iii.
EMA200 ±0.5% 밴드 — 닿음 vs 돌파
abs(close - EMA(close,200)) / close × 100 ≤ 0.5. 가격이 200일 이평선에서 ±0.5% 안쪽으로 들어왔을 때만 "닿음"으로 판정. 0.5%는 BTC 일봉 변동성의 ~1/3 — 너무 좁으면 노이즈, 너무 넓으면 의미 희석. ATR 기반으로 동적 변경도 가능 (band_pct = ATR(14) × 0.3 / close × 100).
tuning ▸ atr-adaptive
05ENGINE
module 5 · backtest engine

M5 백테스트 엔진 — event-loop 시뮬레이션 + run_id 재현성

[ Strategy DSL ] [ Historical OHLCV (M8) ] └──────────┐ ┌────────────────┘ ▼ ▼ ┌─────────────────────┐ │ Backtest Engine │ │ (event-loop sim) │ └─────────────────────┘ │ ┌──────────────┼──────────────┐ ▼ ▼ ▼ Trades list Performance run_id (Sharpe/MDD/ (재현성 해시) 승률 등) ▸ 핵심 규칙 (NO LOOKAHEAD) · 신호 = bar close 이후 확정 · 체결 = 다음 bar open (기본) / intrabar 옵션 · run_id = hash(strategy + data + code_git_sha + seed + fee/slippage_model) └─ 같은 입력이면 항상 같은 결과 보장
i.
No-Lookahead 위반 — 흔한 함정
초보 백테스터의 70%가 저지르는 실수: signal[t] = close[t] > MA[t] 로 신호 만들고 fill[t] = close[t] 로 체결. close는 t 봉이 끝난 뒤에야 확정되는데 같은 봉 close에 체결한 셈이라 미래 정보를 쓴 것. 정답은 fill[t+1] = open[t+1]. Sharpe가 1.5 → 0.3 으로 추락합니다.
bug ▸ time travel
ii.
run_id — 8자리 해시로 재현성 보장
run_id = sha256(yaml + ohlcv_hash + git_sha + seed + fee_model)[:8]. 결과는 a3f9c2b1. 6개월 뒤 같은 yaml로 돌리면 같은 run_id. 다르면 데이터·코드 중 하나가 바뀐 것 → diff 추적. 논문 reproducibility 기준 충족.
trace ▸ deterministic
iii.
Binance 수수료 — 현실적 차감
기본 maker 0.1% / taker 0.1%. BNB 결제 시 25% 할인 → 0.075%. VIP 1티어부터 0.09/0.10%. 슬리피지는 BPS 모델: BTC 5bps, 알트 20bps, 저유동성 50bps. 1년 100회 매매면 수수료만 20% 차감 — 백테스트에서 "잘 되더라" 했다가 실거래 -10% 나는 주범.
fees ▸ realistic model

ii.

guard ▸ circuit breaker
iii.
상위 N — 30 vs 100 백테스트 차이
상위 30: BTC/ETH/메이저 알트만, 슬리피지 평균 5bps, 백테스트 깔끔. 상위 100: 잡코인 포함, 슬리피지 30~50bps, 신호 빈도 3배 ↑ 하지만 펌프&덤프 노이즈. 권장: 백테스트는 30/50/100 셋 다 돌리고 Sharpe·MDD 비교 후 결정.

i.
왜 M8을 첫 번째에 — 데이터 없으면 다 무용
백테스트(M5)·시그널(M4)·차트(M10) 모두 OHLCV가 입력. M8 없이 M5부터 만들면 더미 CSV로 시작 → 나중에 실데이터 붙일 때 인터페이스 깨짐. M8 먼저 만들고 get_ohlcv(sym, tf, start, end) 표준 함수 박아놓으면 그 뒤는 모두 의존성 따라 자연 정렬.
order ▸ data-first
ii.
M5 → M6 — Optimizer가 백테스트를 1만 번 호출
Optuna는 params 공간을 베이지안 탐색으로 1,000~10,000회 sweep. 각 시도가 M5 백테스트 1회. M5가 5년치 백테스트 30초 걸리면 1만 회 = 83시간. 그래서 M5 최적화(numba JIT, vectorize) 필수 → 5년치 1초로 단축 → 10,000회가 3시간.
perf ▸ numba jit
iii.
실거래 직전 — 페이퍼 30일
M9 Binance 어댑터 페이퍼 모드: 실API에서 가격 받지만 주문은 가짜. 30일 페이퍼 운용 → 슬리피지 실측, 신호↔체결 지연 측정, 리스크 가드레일 발동 횟수 체크. 페이퍼 결과가 백테스트 ±20% 이내일 때만 실거래 전환. 차이 크면 Look-Ahead Bias 의심.
stage ▸ paper trading
